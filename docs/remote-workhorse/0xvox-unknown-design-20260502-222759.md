## Problem Statement

The goal is to ship a remote environment that incorporates the user's whole agent workflow instead of becoming another machine to babysit.

The user gives an idea. The environment must:

1. Read the newest source of truth.
2. Select the correct configured tools.
3. Use Research Vault and other memory/research tools when relevant.
4. Produce an explicit repair card when a required tool is unavailable.
5. Route the idea through the established workflow.
6. Run bounded work.
7. Produce verifiable evidence.
8. Close as `PASS`, `FLAG`, or `BLOCK`.

The concrete failure to prevent: an agent says "Research Vault did not work, so I skipped it" or "the remote is configured" without proving host, repo, tool state, run id, exit status, and artifacts.

## What Makes This Cool

This turns the remote from infrastructure into a thinking and execution organ. The "whoa" moment is not a dashboard. It is this interaction:

```text
user: here is an idea
remote: I checked the live issue/repo/MCP/runtime truth, used these tools,
        repaired or flagged these missing surfaces, ran this bounded workflow,
        and here is the evidence-backed verdict.
```

That is the leverage. The user gets to keep throwing messy ideas at the system while the system reliably chooses the right memory, research, review, runtime, and evidence paths.

## Constraints

1. No implementation work in this `/office-hours` pass. This document defines the ship path.
2. Do not copy local desktop config blindly to the remote.
3. Do not expose unrestricted shell MCP.
4. Do not expose filesystem roots like `/`, `$HOME`, `.ssh`, `.config`, or secret paths.
5. Do not put private browser cookies, iMessage, social, or email MCPs on the remote by default.
6. Do not enable cloud-provider or payment writes without operator approval.
7. Do not treat an enabled MCP as usable until health, auth, scope, and target fit are proven.
8. Workbench remains durable operating memory and public-safe loop record; it is not the live scheduler.
9. systemd owns scheduling; runtime-control exposes safe operations and evidence lookup.
10. Research Vault must be checked or explicitly repair-carded.

## Premises

1. The ultimate goal is not "a better server"; it is a reliable per-idea workflow execution center.
2. The first shippable slice should be the workflow contract, tool truth snapshot, and one read-only canary, not full NixOS provisioning.
3. Tool selection must be profile-driven. A mega-profile will recreate local clutter on a stronger machine.
4. Research Vault is a required context source for idea/spec/research/review work when relevant.
5. `code-review-graph` must be graph-bootstrap-gated because current proof shows zero registered repositories.
6. Disabled or unauthenticated tools are not failures by themselves. Silent omission is the failure.
7. Remote success requires evidence from the remote host, not UI state or agent self-report.

## Approaches Considered

### Approach A: Contract-First Canary

Summary: Ship the smallest real workflow first: `IDEA_RUN_CARD`, `SELF_AWARENESS_BOOTSTRAP`, `TOOL_INVENTORY.json`, `RESEARCH_VAULT_PROOF.json`, `CODE_REVIEW_GRAPH_PROOF.json`, and one read-only repo/review health smoke.

Effort: M  
Risk: Low

Pros:

1. Ships the actual behavior change: no silent tool skips, no proofless completion.
2. Avoids provisioning risk while still producing a repeatable workflow.
3. Uses current tools immediately: `codex mcp list`, Research Vault health/taxonomy/search, graph repo list, GitHub/docs checks.
4. Produces artifacts downstream agents can reuse.

Cons:

1. Does not yet make the remote host a full NixOS workhorse.
2. Some checks will initially be local or mock-remote until the first canary host exists.
3. Requires discipline to not treat the contract as "done" before a real canary run proves it.

Reuses:

1. Existing `remote-workhorse-incorporate-everything-spec.md`.
2. Current Codex MCP inventory.
3. Current Research Vault MCP.
4. Workbench self-awareness and PASS/FLAG/BLOCK conventions.

### Approach B: Full NixOS Workhorse Cell

Summary: Provision the NixOS control droplet and build the full runtime cell: `/srv` layout, `sops-nix`, Tailscale/private ingress, `runtimectl`, runtime-control MCP, filesystem_scoped MCP, tool registry, Research Vault bridge, journald evidence, and systemd-run job execution.

Effort: XL  
Risk: Medium

Pros:

1. Lands the final architecture directly.
2. Separates laptop-local tools from remote runtime truth.
3. Makes host/ref/worktree/run/artifact proof a native part of execution.
4. Creates the right foundation for future Slock/acpx/Flue integration.

Cons:

1. Too much surface area for the first ship.
2. Cloud provisioning, secrets, private networking, and MCP policy can mask whether the workflow itself is correct.
3. Higher chance of recreating local config drift before the contract is proven.

Reuses:

1. NixOS/Rust research note.
2. `/Users/0xvox/Desktop/tool.txt` toolchain decisions.
3. Existing NixOS/Rust stack candidates: `disko`, `nixos-anywhere`, `sops-nix`, `rust-overlay`, `crane`, `sccache`, `nextest`, `deny`, `audit`, `machete`, `llvm-cov`, `bacon`.

### Approach C: Front-Door First With Slock/acpx/Flue

Summary: Treat Slock/acpx/Flue as the primary front door and build structured agent-session control first, then attach the remote runtime cell behind it.

Effort: L  
Risk: High

Pros:

1. Attacks the user's visible coordination pain directly: session state, queue visibility, typed status, and handoff.
2. Could replace PTY scraping and make multi-agent sessions easier to operate.
3. Fits the Workbench direction if the backend evidence layer is already stable.

Cons:

1. It optimizes the control surface before proving the execution contract.
2. It risks creating another dashboard to babysit.
3. It can hide missing tool truth behind a nicer UI.

Reuses:

1. Parked `acpx` spike.
2. Slock daemon/runtime visibility.
3. Workbench Flue harness direction.

## Recommended Approach

Choose Approach A now.

The first true ship is not a droplet. The first true ship is a run that proves:

```text
given idea -> checked source of truth -> checked tools -> used or repair-carded Research Vault
           -> blocked graph until repo registered -> selected route
           -> produced reusable evidence -> emitted PASS/FLAG/BLOCK
```

After that run exists, Approach B becomes much safer because the NixOS workhorse has a contract to satisfy. Approach C should wait until the evidence skeleton exists, otherwise it becomes a nicer way to watch an unreliable process.

## Success Criteria

Phase 1 is done only when:

1. A new agent can read the canary report and rerun the same workflow.
2. Research Vault proof is present or an `RV_REPAIR_CARD` exists.
3. Disabled tools are listed as disabled by design.
4. Unauthenticated tools are listed with repair actions.
5. Graph state is not overclaimed.
6. The canary report includes host, cwd, repo/ref if applicable, commands/tool calls, exit state, artifacts, and final verdict.
7. The workflow can say `BLOCK` without shame when a required source of truth is missing.
